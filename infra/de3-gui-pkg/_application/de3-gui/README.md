# Home Lab GUI

A Python [Reflex](https://reflex.dev) web app that visualises the home lab
infrastructure as a tree and provides tooling for running waves, viewing
config, and exploring the infra graph.

## How it works

The app scans `infra/` at startup and builds an in-memory node tree that mirrors
the directory hierarchy. No DAG file is written — the tree is always derived
fresh from the source.

### Configuration

All app settings live in `infra/de3-gui-pkg/_config/de3-gui-pkg.yaml` (outside
the app directory) under the `de3-gui-pkg:` top-level key. Key sections:

| Section | Purpose |
|---------|---------|
| `config` | `vm_ip`, `ansible_inventory_path`, wave log settings |
| `supported` | Canvas views and provider definitions |
| `options` | Default UI state (canvas, provider visibility) |
| `file_viewer` | Editor list, quote-path toggle |
| `unit_params` | Identity param ordering |
| `testing.ui_testing` | Headless mode, observation mode, timeouts, colors |
| `provider_actions` | Context-menu action definitions |

### State persistence

`state/current.yaml` stores the active UI state (expanded nodes, selected node,
filter settings) across restarts. It is written by the app at runtime and is
safe to delete to reset to defaults.

## Running

```bash
cd infra/de3-gui-pkg/_applications/de3-gui

./run --deps     # no-op: venv + requirements are set up automatically on every invocation
./run --build    # reflex init (first time only)
./run --run      # start the server at http://localhost:9080
./run --test     # run the Playwright smoke test
./run --status   # check if app is responding
```

Or via Make:

```bash
make deps    # no-op: venv + requirements are set up automatically on every invocation
make build   # reflex init
make run     # start server
make test    # smoke test
```

## Testing

`make test` / `./run --test`:

1. Checks the backend API at `:9000/api/infra-graph` (waits up to 180 s on cold start).
2. Checks the frontend at `:9080`.
3. Runs the Playwright smoke test via Ansible (`tests/playbooks/smoke-test.yml`).

The smoke test opens a browser, waits for the infra tree to load, and checks that
no "Connection Error" banner is visible. Observation mode, timeouts, and tint colors
are all configurable in `de3-gui-pkg.yaml` under `testing.ui_testing`.

## Explorer views

The top-left panel switches between five explorer roots via the dropdown:

| Root | Description |
|------|-------------|
| **Infra** | Terragrunt unit tree grouped by package and wave |
| **Modules** | Terraform module definitions (`_modules/`) |
| **Packages** | Package list with capability and sync status |
| **Ansible Inventory** | Dynamic inventory grouped by role and wave |
| **Framework Repos** | Cytoscape graph of framework package repositories |

### Framework Repos view

Reads `config/tmp/fw_repos_diagram_exporter/known-fw-repos.yaml` (generated by
`fw-repos-diagram-exporter --list`). Rendered as a Mermaid class diagram inside an
iframe (`assets/fw_repos_mermaid_viewer.html`). Each repo is a class node showing
its source type, git URL, main package, labels, backend, and embedded/external packages.
Arrow edges show "creates" relationships between repos.

Menu bar controls:
- **Refresh** — re-runs `fw-repos-diagram-exporter --list` and reloads data
- **Git ▾** / **Config Pkg ▾** / **Labels ▾** / **Backend ▾** / **Packages ▾** — show/hide diagram sections
- **Inaccessible ▾** — colour unreachable repos red, grey, or hide them
- **Zoom** — scale slider (also scroll-wheel to zoom, double-click to reset)

Canvas interaction:
- **Left-click drag** — pan the canvas
- **Right-click a repo node** — context menu with two actions:
  - **Open Git URL** — opens the repo's browse URL in a new tab (greyed out for local-only repos)
  - **Open framework_package_repositories.yaml** — opens the package's repositories config file in the repo (greyed out when `main_package` is not set)

## Layout modes

### Normal mode (default)
Four-panel fixed layout: infra tree (top-left), file viewer (top-right), terminal (bottom-left), object viewer (bottom-right).

### Floating panels mode
Enable via **Appearance → Layout → Floating panels mode**. The infra tree stays in the left sidebar; the other three panels become independent draggable/resizable floating windows. A **Panels ▾** button appears in the toolbar to show/hide individual panels. Panel positions are saved to `state/current.yaml` on close and restored on next load.

## Appearance menu

The Appearance dropdown is organised into 10 collapsible accordion sections:
**Show in controls bar**, **Infra tree**, **Wave panel**, **File viewer**, **Unit detail popup**, **Terminal**, **Params panel**, **Nested networks**, **Layout**, **Theme & App**.
Sections open/close by clicking the section header. State is persisted across restarts.

## Build status dots

When **Show build status** is enabled (Appearance menu → Infra tree), a coloured 7 px dot appears
before each leaf unit name in the infra tree:

| Colour | Status | Meaning |
|--------|--------|---------|
| Green | `ok` | Resources exist in state |
| Purple | `destroyed` | State file exists but resources = [] (cleanly destroyed) |
| Red | `fail` | Apply failed (non-zero exit code, or in wave queue with no state) |
| Grey | `none` | No state found anywhere — never built or fully cleaned |

### How status is determined

**Tier 0 — exit-status YAMLs (primary, real-time)**

`root.hcl` fires an `after_hook` after every apply/destroy that runs
`utilities/tg-scripts/write-exit-status/run`. This writes a one-shot YAML to
`$_DYNAMIC_DIR/unit-status/exit-<unit>.yaml` and also mirrors the result to GCS
(`unit_status/<unit_path>/<ts>.json`). The `local_state_watcher` loop consumes the
file (deletes it) on next poll and updates the dot immediately — no GCS round-trip
needed for the real-time update.

**Tier 0b — MaaS intermediate status (live progress)**

During long MaaS operations (commissioning, deploying), the MaaS polling scripts write
`$_DYNAMIC_DIR/unit-status/maas-<unit>.yaml` every ~10–30 s. The watcher reads these
without consuming them, updating `maas_phase` / `maas_message` in the hover popup.
Cleaned up by `write-exit-status/run` when the apply completes.

**Tier 3 — apply exit code (GUI-initiated applies)**

When **Apply unit** is used from the context menu, the exit code is written to
`$_GUI_DIR/homelab_gui_apply_<unit>.exit`. The watcher consumes this within one poll
interval: exit code ≠ 0 → `fail`.

**Validate (on demand) — GCS state bucket scan**

Triggered by "⟳ Refresh status". Uses `gsutil ls -l -r` across all `_stack/` prefixes.
Mtime-cached so only changed state files are downloaded. Parses `resources[]` count →
`ok` / `destroyed`. Authoritative for the full historical picture.

**GCS startup sync**

On load, two background tasks (`sync_unit_status_from_gcs`, `sync_wave_status_from_gcs`)
pull prior-session status from GCS (`unit_status/` and `wave_status/` prefixes),
recovering dots and wave history from before the current session.

## File structure

```
de3-gui/
├── homelab_gui/
│   └── homelab_gui.py   # Reflex app — state, computed vars, UI components (~8000 lines)
├── state/
│   └── current.yaml     # Live UI state (written at runtime; safe to delete)
├── tests/
│   ├── browser_test.py  # Playwright DOM assertions
│   ├── playbooks/
│   │   └── smoke-test.yml
│   ├── inventory/
│   └── run              # Test runner (--deps / --test)
├── assets/              # Static HTML viewers (Cytoscape, mxGraph iframes)
├── docs/
│   └── ai-log/          # Per-session change logs
├── rxconfig.py          # Reflex config (reads HOMELAB_GUI_FRONTEND/BACKEND_PORT; defaults 9080/9000)
├── requirements.txt
├── Makefile
├── run                  # Build / run / test script
└── CLAUDE.md            # AI session instructions
```
